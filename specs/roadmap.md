## Phase 1: Action MVP
**Status**: Complete — landed 2026-04-21 (Wave 3 pivot)
**Goal**: Ship the reviewer as a Docker-container GitHub Action that a maintainer can add to any repo with a one-line `uses:` and have it review the next PR.

> _Scope evolution (2026-04-21, Wave 3): the MVP originally targeted a self-hosted FastAPI webhook server. That code shipped in Waves 1 + 2 and was **deleted** in Wave 3 when we pivoted to the Action. `src/main.py`, the HMAC signature verification, `BackgroundTasks` queue, and all `/webhook/*` / `/review/*` / `/analyze/*` / `/health` endpoints are gone. Target users (OSS maintainers + 5-50 dev teams) want zero-hosting install; the Action delivers that. GitLab support went with it — see Phase 4._

- [x] Action metadata (`action.yml`) — inputs: `github-token`, `openai-api-key`, `moonshot-api-key`, `llm-provider`, `llm-model`, `llm-base-url`, `max-files-to-review`, `max-diff-lines`, `review-enabled`, `dry-run`.
- [x] `Dockerfile` — `python:3.13-slim`, pip-installs `src/requirements.txt`, entrypoint `python -m src.action`.
- [x] Action entry point (`src/action.py`) — maps `INPUT_*` env vars onto `Settings` names, parses `GITHUB_EVENT_PATH`, only triggers on `pull_request` with actions `opened` / `synchronize` / `reopened`, calls `PRReviewerAgent.review_pr`. `CLI_REPO` / `CLI_PR` fallback for local smoke-tests.
- [x] GitHub API client (`src/tools/github_tools.py`) — PyGithub-backed. `get_pr_details` fetches metadata + file patches (truncated at 2000 chars/file), applies `partition_files` (lockfiles / vendored / generated / minified dropped), applies secret redaction on each patch. `post_review_comment` creates a COMMENT-type PR review; the tool layer short-circuits on `INPUT_DRY_RUN=true`.
- [x] Core review agent (`src/agents/pr_reviewer.py`) — LangChain `create_openai_tools_agent`, `max_iterations=5`, `temperature=0.1`, repo-agnostic system prompt (kept bit-identical for prompt-prefix caching).
- [x] Configuration system (`src/utils/config.py`) — `pydantic-settings`-backed. `LLM_PROVIDER` + per-provider key validator, size limits, kill switch.
- [x] Error handling / logging — stdlib `logging`, structured one-line format; usage totals logged after each run.
- [x] Tests — pytest suite (~107 tests) covering config, LLM factory, diff filter, guardrails, redactor, and the Action entry point.

**Definition of Done**: `uses: owner/repo@main` in a workflow file produces a posted review on a test PR; `action.yml` + `Dockerfile` + `src/action.py` exist and are wired end-to-end; `pytest -q` is green; `docker build -t prb:test .` succeeds from repo root.
**Met**: all four. Real-PR smoke-test is still pending per-provider (see Phase 1.6).

## Phase 1.5: Safety & Cost Guardrails
**Status**: Complete except for retry/backoff + tiktoken budget (both explicitly deferred to a later wave)
**Goal**: Prevent runaway costs and noisy reviews before layering more intelligence on top

- [x] Size limits — `MAX_FILES_TO_REVIEW` and `MAX_DIFF_LINES` are read in `PRReviewerAgent.review_pr`. Over-limit posts a polite "too large" notice via `post_review_comment` and does NOT invoke the agent.
- [x] Skip list for low-value files (`src/utils/diff_filter.py`) — applied inside `GitHubTools.get_pr_details`:
  - [x] Drafts / `draft: true` PRs
  - [x] Auto-generated markers (`@generated`, `AUTOGENERATED`, `DO NOT EDIT`, `Code generated by`) — first ~20 added lines
  - [x] Lockfiles: `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `npm-shrinkwrap.json`, `poetry.lock`, `Pipfile.lock`, `uv.lock`, `Cargo.lock`, `Gemfile.lock`, `composer.lock`, `go.sum`
  - [x] Vendored paths: `vendor/`, `node_modules/`, `dist/`, `build/`, `.next/`, `__pycache__/`, `.venv/`, `venv/` (segment-aware)
  - [x] Minified assets: `.min.js`, `.min.css`
- [x] Kill switch — `REVIEW_ENABLED=false` makes `review_pr` return `{success: True, skipped: "disabled"}` without fetching or calling the LLM.
- [x] **Secret redaction — LANDED 2026-04-21 (Wave 3)**. `src/utils/redactor.py` scrubs GitHub tokens, OpenAI keys, Slack tokens, JWTs, AWS access keys, AWS secrets (value only, prefix preserved), PEM private-key blocks, and generic `password|secret|api_key|auth_token = "..."` assignments. Applied inside `GitHubTools.get_pr_details` BEFORE the data reaches the agent. Counts are logged; matches are never logged.
- [ ] Retries with backoff on GitHub / OpenAI / Moonshot 5xx + rate-limit responses — **STILL PENDING**.
- [ ] Per-PR token budget via tiktoken — **STILL PENDING**.

**Short-circuit order in `review_pr`** (Wave 3 shape — each returns before the LLM is called):
1. Kill switch (`review_enabled`)
2. Fetch details
3. Draft / WIP
4. Size limits (posts a polite "too large" notice)
5. Otherwise → invoke agent with a `UsageCallbackHandler`, log usage totals at INFO, return.

> _Wave 3 removed step (2) "durable dedup" because the Action is ephemeral. GitHub's own event semantics handle dedup (the workflow only triggers on `opened` / `synchronize` / `reopened`). No `history.has_review` lookup remains._

**Definition of Done**: A 10k-line PR, a vendored-only PR, and a draft PR all result in either no LLM call or a bounded one. Secrets that leak into a patch never reach the LLM. Running the Action over a week of real traffic incurs predictable, bounded cost.
**Met**: All except retry/backoff and tiktoken budget.

## Phase 1.6: LLM Provider Abstraction (OpenAI + Kimi)
**Status**: Code complete 2026-04-21; real-PR smoke test on Kimi still pending
**Goal**: Run the review agent against OpenAI **or** Moonshot's Kimi (K2.5 / K2.6) with a single config switch, and leave a clean extension point for future providers.

### Design (landed)

- [x] **Config**: `LLM_PROVIDER` env var (`openai` | `kimi`), `LLM_MODEL` override (alias: legacy `OPENAI_MODEL`), `LLM_BASE_URL` override, `MOONSHOT_API_KEY`. Cross-field validator fails fast if the selected provider's key is missing.
- [x] **Factory** (`src/utils/llm.py`): `get_chat_model(settings)` returns a `ChatOpenAI` configured for OpenAI or Moonshot. Defaults: `gpt-4o-mini` / `kimi-k2-0905-preview`. `LLM_BASE_URL` overrides either. Passes `user="pr-reviewer-agent"` via `model_kwargs` so OpenAI's automatic prefix cache is scoped per app.
- [x] **Agent wiring** (`src/agents/pr_reviewer.py`): uses `get_chat_model(settings)` instead of constructing `ChatOpenAI` directly.
- [x] **Env template** (`src/.env.example`): two-block "choose ONE provider" structure.
- [x] **Docs**: `specs/tech-stack.md` lists Moonshot as a supported provider.
- [x] **Tests**: config tests (key-missing errors per provider; alias `OPENAI_MODEL` still works); factory tests (correct model + base URL per provider; no live API).

### Explicit non-goals (unchanged from 2026-04-21)
- Streaming responses (not needed for background review).
- Provider-native caching beyond the passive `user` identifier (deferred).
- Non-OpenAI-compatible providers (Anthropic, Gemini, Ollama).
- Kimi's `kimi-k2-thinking*` reasoning modes.
- Auto-failover between providers.

**Definition of Done**: `LLM_PROVIDER=openai` and `LLM_PROVIDER=kimi` both produce a posted review on a test PR with no code changes beyond the workflow input. Tests cover both factory paths. Tech-stack doc and env template reflect the new knobs.
**Met so far**: Code + tests. The two-provider real-PR smoke test is still pending (gate for Definition of Done).

## Phase 1.7: SQLite History KV + Prompt Caching
**Status**: DROPPED 2026-04-21 (Wave 3)

Dropped when we pivoted to the Action delivery model. Actions are ephemeral — each run is a fresh container with no writable durable state — so a durable KV has no value. GitHub's own event semantics (the workflow triggers only on `opened` / `synchronize` / `reopened`) cover the dedup story that the KV was meant to handle.

What WAS built and then removed (Wave 3 delete list): `src/utils/history.py`, `tests/test_history.py`, the four history-backed tests in `tests/test_guardrails.py`, and the `HISTORY_ENABLED` / `HISTORY_DB_PATH` / `HISTORY_RETENTION_DAYS` config fields.

What survives (still on `main`):
- Token-usage capture via `src/utils/usage_callback.py` + `UsageCallbackHandler`. Usage is now **logged** at INFO after each review (`prompt=X completion=Y cached=Z`) instead of persisted.
- The passive per-app prompt-cache identifier (`user="pr-reviewer-agent"` in `get_chat_model`). OpenAI's automatic prefix cache still benefits.

Out of scope for any future phase:
- Per-repo `prompt_cache_key` (requires a stable per-invocation pass-through on LangChain's `AgentExecutor` that 0.3.x doesn't expose).
- Moonshot / Kimi explicit context caching (provider-specific body plumbing; deferred until a concrete ask).

### What actually exists today (as of 2026-04-21, Wave 3)

- `action.yml` — Docker-container Action metadata with 10 inputs.
- `Dockerfile` — `python:3.13-slim`, `pip install -r src/requirements.txt`, `ENTRYPOINT ["python", "-m", "src.action"]`.
- `src/action.py` — Entry point. Maps `INPUT_*` onto `Settings`-compatible env vars, parses `GITHUB_EVENT_PATH`, triggers on pull_request `opened` / `synchronize` / `reopened`, exits 0 for any other event, exits 1 on missing secrets or agent error. Has a `CLI_REPO` / `CLI_PR` local-smoke-test fallback.
- `src/agents/pr_reviewer.py` — GitHub-only `PRReviewerAgent` using `create_openai_tools_agent`, `max_iterations=5`, `temperature=0.1`. Short-circuit order: kill switch → fetch → draft → size limits → agent invoke. Logs aggregated token usage at INFO after each run.
- `src/tools/github_tools.py` — `GitHubTools` (via PyGithub) + `GetPRDetailsTool` / `PostReviewTool`. Applies `partition_files` to drop lockfiles / vendored / generated / minified files, applies `redact_with_count` on each remaining patch, posts a COMMENT-type PR review. `PostReviewTool._run` honors `INPUT_DRY_RUN=true` to skip the actual post.
- `src/utils/config.py` — Trimmed `Settings`: `llm_provider`, `llm_model`, `llm_base_url`, `openai_api_key`, `moonshot_api_key`, `github_token`, `agent_name`, `max_files_to_review`, `max_diff_lines`, `review_enabled`. No more `platform`, `gitlab_*`, `*_webhook_secret`, `api_*`, `history_*`.
- `src/utils/llm.py` — Provider factory (OpenAI or Moonshot/Kimi). Unchanged from Wave 2b.
- `src/utils/diff_filter.py` — Pure helpers (`is_skippable_path`, `is_generated_patch`, `count_diff_lines`, `partition_files`). Unchanged.
- `src/utils/redactor.py` (new Wave 3) — Ordered pattern-based secret scrubber. Returns `[REDACTED:<kind>]` placeholders; exposes a count for logging.
- `src/utils/usage_callback.py` — `UsageCallbackHandler`. Unchanged from Wave 2b; still captures token usage across all LLM calls in a run.
- `tests/` — 107 tests covering config, LLM factory, diff filter, redactor, guardrails (GitHub-only now), usage callback, and the Action entry point.

### Known gaps

- **Retry / backoff on GitHub / OpenAI / Moonshot 5xx + rate-limit responses** — still pending. Plan: `tenacity` or equivalent on `get_pr_details` / `post_review_comment` / the LLM invoke.
- **tiktoken budget estimation** — still pending. Plan: estimate prompt + completion tokens before invoking the agent; refuse or trim over-budget PRs.
- **Kimi real-PR smoke-test** — the code path is covered by unit tests; a live end-to-end run is still required to close Phase 1.6's Definition of Done.
- **No Marketplace listing yet** — Phase 4 will tag v1.0.0, list on Marketplace, and start pushing GHCR images for faster cold start.
- **GHCR image build workflow** — not implemented. Currently every Action consumer rebuilds the image from Dockerfile at first use. Phase 4.
- **Diff parser / inline comments** — LANDED 2026-04-21 (Wave 4). `src/utils/diff_parser.py` parses unified-diff patches into `DiffLine` entries with per-line `position` values; `PostReviewTool` posts review-comment batches via `pr.create_review(comments=[...])`. Comments whose line falls outside the diff window are skipped (count logged, body never logged).
- **Logging is stdlib plain-text** — no structured JSON, no request/run IDs. Fine for the Action console but would need to change if we ever ship a hosted dashboard.
- **Secret redaction is regex-based** — high-recall on the listed patterns but not exhaustive. Things we don't catch today: Azure connection strings, Google service-account JSONs, Stripe keys (`sk_live_...` clashes with our OpenAI `sk-` pattern for short suffixes), private npm tokens. Expand as we see real traffic.
